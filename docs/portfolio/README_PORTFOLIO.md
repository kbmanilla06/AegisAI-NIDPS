# AegisAI NIDPS — Synthetic IDS Portfolio Demonstration

## What this project demonstrates

AegisAI NIDPS is a defensive, portfolio-oriented network intrusion-detection
platform. The published first release demonstrates controlled telemetry
normalization, deterministic evidence, synthetic offline model/anomaly
evidence, analyst workflows, aggregate monitoring, and prevention simulation.

It is not a production IPS, a live network monitor, or evidence of performance
on real traffic.

> **SYNTHETIC DEMO ONLY.** This dataset and its labels were generated by AegisAI
> scenarios to test software contracts. Results do not measure UNSW-NB15
> performance, real-network intrusion detection, generalization, zero-day
> detection, production readiness, or prevention suitability. The model is
> offline-only and cannot create or modify alerts or prevention actions.

The monitoring surfaces use the corresponding monitoring limitation contract:

> **SYNTHETIC DEMO ONLY.** Monitoring evidence is derived from project-generated
> synthetic metadata and offline artifacts. It does not measure real-network
> drift, UNSW-NB15 performance, production readiness, or prevention suitability.
> Monitoring cannot activate models or create or modify alerts, detections,
> incidents, or prevention actions.

## Safety boundary

- UNSW-NB15 acquisition is blocked and publisher outreach is cancelled.
- No real or third-party dataset bytes are used.
- Live packet capture, public-network deployment, online inference, and model
  activation are absent.
- Prevention is simulation-only and makes no firewall or host-state change.
- The API remains the authorization boundary; RBAC, CSRF/Origin, audit,
  retention, and artifact SHA-256 controls remain in force.

The machine-readable false-capability flags are all `false` for real dataset
use, UNSW acquisition, live capture, online inference, model activation, alert
side effects, and prevention.

## Clean local demonstration

The authoritative procedure is
[`DEMO_RUNBOOK.md`](DEMO_RUNBOOK.md). It uses the existing ARM64 Docker Compose
topology, localhost-only ports, synthetic/offline evidence, and no credentials
or external services. The runbook does not download data or load a model.

```text
1. Copy .env.example to an ignored .env and use validation-only local values.
2. Start the disposable Compose project on localhost.
3. Verify liveness, readiness, dashboard HTTP 200, and Celery health.
4. Run the metadata-only Gate P5 evidence helper.
5. Review the deterministic transcript and aggregate evidence.
6. Tear down the disposable project and remove temporary evidence.
```
